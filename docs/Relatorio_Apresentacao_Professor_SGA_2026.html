SGA - Sistema de Gestão de Serviços Académicos

Unidade Curricular: Desenvolvimento Web / Projeto de Aplicações
Tipo de Documento: Relatório completo para apresentação ao professor
Data: 20/03/2026

1. Introdução

Este relatório apresenta o projeto SGA (Sistema de Gestão de Serviços Académicos), uma aplicação web desenvolvida para apoiar o funcionamento académico de uma instituição de ensino. O sistema foi pensado para centralizar processos importantes, reduzir trabalho manual e organizar a informação de forma segura, clara e acessível para diferentes perfis de utilizadores.

O foco principal da aplicação é permitir que cada perfil (aluno, funcionário e gestor) execute as suas tarefas dentro do mesmo sistema, com permissões adequadas e interface simples de utilizar.

2. Objetivo do Projeto

O objetivo do SGA é disponibilizar uma plataforma única para:

  • Gestão de fichas de aluno.
  • Processamento de pedidos de matrícula/inscrição.
  • Registo e consulta de pautas/notas.
  • Administração de cursos, unidades curriculares e utilizadores.
  • Produção de relatórios de apoio à gestão académica.

3. Funcionalidades da Aplicação

3.1 Funcionalidades Gerais

  • Login seguro por utilizador e palavra-passe.
  • Dashboards diferentes por perfil.
  • Mensagens de feedback (sucesso/erro) em todas as operações principais.
  • Suporte multilingue (Português e Inglês).
  • Tema visual claro/escuro com design consistente.

3.2 Funcionalidades do Aluno

  • Visualizar dashboard pessoal.
  • Criar e editar a ficha de aluno.
  • Acompanhar o estado da ficha (rascunho, submetida, aprovada, rejeitada).
  • Consultar pedidos de matrícula/inscrição e respetivo estado.
  • Usar o módulo de mensagens internas com funcionários.

3.3 Funcionalidades do Funcionário

  • Consultar e gerir pautas de avaliação.
  • Introduzir e atualizar notas por aluno.
  • Consultar pedidos pendentes de matrícula/inscrição.
  • Aprovar ou rejeitar pedidos com observações.
  • Avaliar fichas submetidas pelos alunos.

3.4 Funcionalidades do Gestor

  • Gerir cursos (criar, editar, desativar).
  • Gerir unidades curriculares (criar, editar, desativar).
  • Gerir utilizadores e perfis de acesso.
  • Consultar indicadores globais da plataforma.
  • Exportar relatórios em formato CSV.

6. Como o sistema funciona na prática

6.1 Fluxo principal

  • O utilizador faz login.
  • O sistema identifica o perfil e mostra o dashboard correspondente.
  • As operações permitidas são apresentadas conforme o perfil.
  • Cada ação (guardar, editar, aprovar, rejeitar) é registada na base de dados.
  • O utilizador recebe feedback visual após cada operação.

6.2 Exemplo de ciclo de ficha e pedido

  1. Aluno preenche/atualiza ficha.
  2. Fica registada com estado e data de atualização.
  3. Pedido de matrícula/inscrição é analisado por funcionário/gestor.
  4. Pedido passa para aprovado ou rejeitado, com observações.
  5. Aluno acompanha resultado no dashboard.

7. Como os dados são guardados

A aplicação utiliza base de dados relacional (MySQL/MariaDB), com tabelas separadas por tema: utilizadores, cursos, UCs, fichas, pedidos, pautas e notas. As tabelas estão ligadas entre si por relações (chaves estrangeiras), garantindo coerência dos dados.

  • Dados pessoais e académicos dos alunos ficam em tabelas dedicadas.
  • Estados dos processos (pendente/aprovado/rejeitado) ficam guardados em campos próprios.
  • Datas de criação/atualização permitem histórico de alterações.
  • Acesso aos dados é feito com queries preparadas (maior segurança).

8. Qualidade visual e alinhamento com o estilo da app

A interface do sistema segue um estilo moderno e minimalista, com foco em legibilidade, contraste equilibrado e consistência visual. O relatório mantém essa mesma linha: cores sóbrias, hierarquia clara de títulos, tabelas limpas e linguagem objetiva, facilitando a leitura por docentes e avaliadores.

9. Pontos fortes do projeto

  • Boa organização por camadas (MVC).
  • Separação clara de permissões por perfil.
  • Cobertura funcional ampla para contexto académico.
  • Estrutura de base de dados consistente e preparada para evolução.
  • Suporte a idioma e personalização de tema.

10. Melhorias futuras sugeridas

  • Adicionar proteção CSRF em todos os formulários.
  • Reforçar políticas de segurança no login (limite de tentativas).
  • Padronizar totalmente o roteamento (evitar duplicação entre rotas e páginas diretas).
  • Expandir auditoria de operações críticas.
  • Criar testes automatizados para maior robustez.

11. Conclusão

O SGA cumpre de forma sólida os objetivos propostos para um sistema de gestão académica, oferecendo funcionalidades relevantes para diferentes perfis de utilizador e uma base técnica organizada para continuidade do projeto. A aplicação está apta para apresentação académica, demonstrando integração entre lógica de negócio, interface e persistência de dados.
